# 为什么能用动态规划解决「最长公共子序列」？

## 从一个故事开始

假设你和朋友各有一串珠子，你想知道：**保持顺序不变，最多能串出多长的一条公共项链？**

比如你的珠子是：🔴🟡🟢🔵🟣（abcde），朋友的珠子是：🔴🟢🟣（ace）。

你能选出的最长公共序列是 🔴🟢🟣（a→c→e），长度为 3。

那计算机怎么自动找出这个答案呢？

---

## 一、遇到的新问题 → 能不能变成熟悉的旧问题？

这是动态规划最核心的思维方式：**把大问题拆成小问题，小问题拆成更小的问题，直到答案显而易见。**

### 1.1 我们先从最简单的场景开始想

**问题 A：如果两个字符串都只有一个字符，你能告诉我答案吗？**

`"a"` 和 `"a"` → 一样，长度是 **1**
`"a"` 和 `"b"` → 不一样，长度是 **0**

这太简单了，一眼就能看出来。

---

**问题 B：如果其中一个字符串是空的呢？**

`""` 和 `"abc"` → 空字符串能和任何东西配出什么？什么也配不出，长度是 **0**

这也是显然的。

---

**关键来了**：我们面对的是 `"abcde"` 和 `"ace"` 这种长字符串，不能一眼看出。但我们可以**从短到长，一步步推导出来**。

---

## 二、核心洞察：只看最后一个字符

假设我们手里已经知道两个字符串各自**少一个字符**的答案，那么：

### 情况 1：两个字符串的最后一个字符相同

```
text1 = "abcd[e]"
text2 = "a[c][e]"
         最后一个字符都是 'e'
```

既然末字符相同，这对字符**肯定能配成一对**！于是：

> **答案 = 「都没最后一个字符」的答案 + 1**

用符号写：`dp[i][j] = dp[i-1][j-1] + 1`

就像说：我发现两颗珠子颜色一样，扣在一起，然后看前面还能扣几对。

---

### 情况 2：两个字符串的最后一个字符不同

```
text1 = "abcd[e]"
text2 = "a[c]"
         末字符 'e' ≠ 'c'
```

末字符不同，意味着不可能两个都用上。那我只能：

- **放弃 text1 的末字符**，看剩下的：`"abcd"` vs `"ac"`
- **放弃 text2 的末字符**，看剩下的：`"abcde"` vs `"a"`

然后**取两种情况中较大的**。

> **答案 = max(放弃 text1 末字符的答案, 放弃 text2 末字符的答案)**

用符号写：`dp[i][j] = max(dp[i-1][j], dp[i][j-1])`

就像说：两颗珠子颜色不一样，我只能扔掉一颗，看哪边剩下的能配出更多。

---

## 三、用一张表来「记账」

想象你有一张表格，行是 text1 的字符，列是 text2 的字符：

```
        ""   a    c    e
  ""    0    0    0    0
  a     0    ?    ?    ?
  b     0    ?    ?    ?
  c     0    ?    ?    ?
  d     0    ?    ?    ?
  e     0    ?    ?    ?
```

- 第 0 行、第 0 列全是 `0`，因为空字符串跟谁配都是 0。
- 每个 `?` 代表一个问题，比如第 3 行第 2 列代表：`"abc"` 和 `"ac"` 的答案。

---

### 3.1 怎么填这张表？从左到右、从上到下，像写作业一样！

我们以填 `dp[1][1]`（第一行第一列）为例：

```
比较 text1[0] = 'a' 和 text2[0] = 'a'
它们相同！
→ dp[1][1] = dp[0][0] + 1 = 0 + 1 = 1
```

再填 `dp[1][2]`：

```
比较 text1[0] = 'a' 和 text2[1] = 'c'
它们不同！
→ dp[1][2] = max(dp[0][2], dp[1][1]) = max(0, 1) = 1
```

这样一格一格填，最终填完整张表，右下角就是答案。

---

## 四、完整的填表动画

用例子 `text1 = "abcde"`, `text2 = "ace"` 来走一遍：

### 初始状态

```
        ""   a   c   e
  ""     0   0   0   0
  a      0   ?   ?   ?
  b      0   ?   ?   ?
  c      0   ?   ?   ?
  d      0   ?   ?   ?
  e      0   ?   ?   ?
```

### 填第 1 行（字符 a）

```
'a' vs 'a' → 相同！取左上+1 → 1
'a' vs 'c' → 不同！取 max(上, 左) = max(0, 1) → 1
'a' vs 'e' → 不同！取 max(上, 左) = max(0, 1) → 1

        ""   a   c   e
  ""     0   0   0   0
  a      0   1   1   1   ← 填完了
  b      0   ?   ?   ?
  c      0   ?   ?   ?
  d      0   ?   ?   ?
  e      0   ?   ?   ?
```

### 填第 2 行（字符 b）

```
'b' vs 'a' → 不同！取 max(上, 左) = max(1, 0) → 1
'b' vs 'c' → 不同！取 max(上, 左) = max(1, 1) → 1
'b' vs 'e' → 不同！取 max(上, 左) = max(1, 1) → 1

        ""   a   c   e
  ""     0   0   0   0
  a      0   1   1   1
  b      0   1   1   1   ← 填完了
  c      0   ?   ?   ?
  d      0   ?   ?   ?
  e      0   ?   ?   ?
```

### 填第 3 行（字符 c）

```
'c' vs 'a' → 不同！取 max(上, 左) = max(1, 0) → 1
'c' vs 'c' → 相同！取左上+1 = 1+1 → 2  ✨ 增长了！
'c' vs 'e' → 不同！取 max(上, 左) = max(1, 2) → 2

        ""   a   c   e
  ""     0   0   0   0
  a      0   1   1   1
  b      0   1   1   1
  c      0   1   2   2   ← 填完了
  d      0   ?   ?   ?
  e      0   ?   ?   ?
```

### 填第 4 行（字符 d）

```
'd' vs 'a' → 不同！→ 1
'd' vs 'c' → 不同！→ 2
'd' vs 'e' → 不同！→ 2

        ""   a   c   e
  ""     0   0   0   0
  a      0   1   1   1
  b      0   1   1   1
  c      0   1   2   2
  d      0   1   2   2   ← 填完了
  e      0   ?   ?   ?
```

### 填第 5 行（字符 e）

```
'e' vs 'a' → 不同！→ 1
'e' vs 'c' → 不同！→ 2
'e' vs 'e' → 相同！取左上+1 = 2+1 → 3  ✨ 又增长了！

        ""   a   c   e
  ""     0   0   0   0
  a      0   1   1   1
  b      0   1   1   1
  c      0   1   2   2
  d      0   1   2   2
  e      0   1   2   3   ← 答案！
```

右下角 `3` 就是最终答案。

## 五、为什么这个思想能工作？—— 三个关键

### 关键 1：问题的答案可以从「更小的自己」推导出来

就像玩俄罗斯方块，你不必每次从头算，你只需要知道：
- 如果当前两个字符一样 → 配对！然后看前面配了多少对。
- 如果当前两个字符不一样 → 扔掉一个，看哪种扔法更好。

这叫做 **最优子结构**。

### 关键 2：同样的「小问题」不会重复算

比如 `"ab"` vs `"ac"` 这个子问题可能被很多更大的问题问到。我们用表记下来，**查表就行，不用重算**。

这叫做 **重叠子问题**。

### 关键 3：从最小的、显然的问题出发，逐步扩大

- 空字符串 vs 任何 = 0（显然）
- 单字符 vs 单字符 = 相同则 1，不同则 0（显然）
- 利用这些小答案，推导大答案

这叫做 **自底向上的递推**。

## 六、一个生活中的类比

想象你在对两条 DNA 序列做比对（其实这就是生物信息学中的经典应用！）。

你有一个「比较记录本」（DP 表），从左往右、从上往下，一格一格记录：

> 「到 text1 的这个位置为止，和 text2 的那个位置为止，最长的匹配是多长？」

每填一格，你只关心三个已经填好的邻居：
- **左上角**：两个都少一个字符的答案（用于「配对」场景）
- **正上方**：只少 text1 一个字符的答案（用于「扔掉 text1 字符」场景）
- **正左方**：只少 text2 一个字符的答案（用于「扔掉 text2 字符」场景）

你需要做的就是：**查这三个数，然后做一个简单的 if 判断**。

## 七、新手常见的困惑，我来一一解答

### Q1：为什么要从左上角+1？为什么不是从上面或左边+1？

因为 **一个字符只能匹配一次**。`dp[i-1][j-1]` 代表的是「两个字符串都去掉末字符」之后的最大匹配数，现在末字符相同，恰好可以加上这对新匹配，所以 +1。

如果你从上边或左边 +1，就意味着你要么说 text1 的这个字符匹配了两次，要么说 text2 的这个字符匹配了两次，这不对。

### Q2：为什么取 max(上面, 左边)，而不是加它们？

因为末字符不同时，你只能**放弃其中一个**，不能两个都用。所以你比较两种放弃方式的优劣。

你不可能把上面和左边的值加起来——那意味着你把 text1 的第 i 个字符同时既用了又不用的矛盾状态。

### Q3：这跟暴力枚举有什么区别？为什么更快？

暴力枚举：你需要枚举 text1 的所有子序列（有 2^m 个！），然后逐个检查是不是 text2 的子序列。当字符串长度为 100 时，2^100 是个天文数字。

DP：只填一张 m×n 的表格，做 m×n 次简单比较。100×100 = 10000 次操作，秒出答案。

**DP 的精髓就是把指数级问题变成多项式级问题。**

### Q4：我怎么知道一个问题能不能用 DP 解决？

问自己三个问题：
1. **能不能拆？** 大问题的答案能不能由小问题的答案拼出来？
2. **有没有重叠？** 同样的小问题会不会被反复遇到？
3. **最优解是否包含最优子解？** 最终答案的每个部分，本身是不是也是最优的？

如果三个问题都回答「是」，那大概率可以用 DP。

---

## 八、一步步教你「遇到新题怎么想到用 DP」

假设你从没学过这个算法，要做这道题。你的思考过程应该是：

1. **先试暴力**：枚举 text1 所有子序列，去 text2 里匹配。发现太慢，不可行。

2. **观察重复**：你注意到 `"abc"` vs `"ac"` 的答案，在很多不同的枚举分支里都会被问到。那不如把它记下来。

3. **找最小问题**：空字符串的答案是 0，这是出发点。

4. **找递推关系**：从 `"ab"` vs `"ac"` 的答案，怎么推出 `"abc"` vs `"ac"` 的答案？观察末字符决定。

5. **定义状态和填表**：定义 `dp[i][j]`，找到转移方程，从 (0,0) 往 (m,n) 填。

6. **答案在右下角**，返回。

---

## 九、一句话总结

> **动态规划不是魔法，它是把大问题变成小问题、把不知道的变成已知的、把算过的记下来不再算第二遍的朴素智慧。**

你不需要一次性理解所有细节。先用例子手动画几张 DP 表，感受一下「从左到右、从上到下填表」的过程，你会发现：**每一格的计算都只需要做一次简单的比较，而整个表格填完，复杂的答案就自然出现了。**

这就是动态规划的美妙之处。
